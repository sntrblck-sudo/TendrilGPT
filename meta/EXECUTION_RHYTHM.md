# Execution Rhythm

This file defines the smallest recurring rhythm for keeping TendrilGPT in motion without creating noise.

## Purpose

Create a lightweight cadence that regularly turns priorities into concrete repo-side moves.

## Core rule

At each rhythm checkpoint, prefer one small completed move over a larger plan.

## Minimal cadence

### Daily or active-session checkpoint
Ask:
- What is the current live item?
- What is the smallest concrete next move?
- Does it belong in GitHub now?

If yes, promote it into the repo immediately.

### Recovery checkpoint
Ask:
- What was the last durable checkpoint?
- Was the interruption stalled, broken, partial, or blocked?
- What is the smallest safe corrective move?

If action is taken, record it in `meta/RECOVERY_LOG.md` when appropriate.

### Queue checkpoint
Ask:
- Which queued item should move next?
- Is there a smaller version that can be completed now?
- Should any item be marked blocked, done, or superseded?

Update `meta/ACTIVE_QUEUE.md` with minimal edits.

## Healthy weekly pattern

A strong week does not require many changes. It requires:
- at least one concrete repo-side move
- at least one queue update when priorities change
- recovery notes only when something actually breaks or stalls
- no unnecessary duplication between repo, Notion, and chat

## Anti-noise rule

Do not create artifacts just to maintain activity.
The rhythm exists to keep real work flowing, not to simulate progress.

## First use

Initial operating sequence:
1. establish repo live-state
2. define recovery drill
3. log recovery actions
4. define active queue
5. establish execution rhythm
