# Issue Rhythm

This file explains when a live thread or idea should become a GitHub issue.

The goal is not to create many issues.
The goal is to turn real implementation pressure into a legible queue.

---

## Use an issue when

Create an issue when something is:
- concrete enough to act on
- likely to require more than one step
- worth tracking across sessions
- easier to discuss as a unit of work than as a floating idea

Good examples:
- implement a small feature
- clean up a recurring repo weakness
- add a missing template or file
- connect two tools or workflows
- test a specific system improvement

---

## Do not use an issue when

Do not create an issue for:
- vague mood or direction
- raw brainstorming
- identity observations that are still forming
- signals that have not yet become actionable
- tiny edits that can just be made directly

Those belong in chat, a note, a signal, or a live thread.

---

## Good issue shape

A useful issue should usually contain:
- a clear title
- why it matters
- what done looks like
- the smallest next move

If any of those are missing, the idea may still be too soft.

---

## Preferred cadence

Use issues lightly.
A few live issues are better than a long graveyard.

A good rhythm is:
1. notice a recurring need
2. test or clarify it a little
3. open an issue when the work becomes concrete
4. close it when the repo is measurably better

---

## Rule

Issues should hold implementation pressure.
Threads hold open questions.
Signals hold notable things.
Experiments reduce uncertainty.
Prompts preserve reusable instructions.
